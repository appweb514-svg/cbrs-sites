# Back-end CMS CBRS Implementation Plan

> **For Hermes:** Use subagent-driven-development skill to implement this plan task-by-task.

**Goal:** Préparer puis construire un back-end complet pour administrer le site CBRS avec comptes administrateur et formateur, modification des activités, flash info et paramètres visuels du site.

**Architecture:** Garder le site public simple et statique côté visiteur, puis ajouter un petit back-end local/serveur pour gérer les contenus. Le back-end expose une API JSON et une interface `/admin` sécurisée. Les données éditables remplacent progressivement les contenus codés en dur dans les fichiers HTML.

**Tech Stack recommandé:** Node.js + Express + SQLite + sessions sécurisées + bcrypt. Front admin en HTML/CSS/JS simple, sans framework lourd. Déploiement possible ensuite sur VPS, Render, Railway, Fly.io ou équivalent.

## 4. Modèle de données SQLite

### Table `users`

```sql
CREATE TABLE users (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  email TEXT NOT NULL UNIQUE,
  password_hash TEXT NOT NULL,
  role TEXT NOT NULL CHECK(role IN ('admin', 'formateur')),
  name TEXT NOT NULL,
  active INTEGER NOT NULL DEFAULT 1,
  created_at TEXT NOT NULL DEFAULT CURRENT_TIMESTAMP,
  updated_at TEXT NOT NULL DEFAULT CURRENT_TIMESTAMP
);
```

### Table `activities`

```sql
CREATE TABLE activities (
  id TEXT PRIMARY KEY,
  name TEXT NOT NULL,
  slug TEXT NOT NULL UNIQUE,
  category TEXT,
  logo TEXT,
  short_description TEXT,
  presentation TEXT,
  meeting_point TEXT,
  level TEXT DEFAULT 'Tous niveaux',
  practical_info TEXT DEFAULT '[]',
  map_lat REAL,
  map_lon REAL,
  published INTEGER NOT NULL DEFAULT 1,
  updated_at TEXT NOT NULL DEFAULT CURRENT_TIMESTAMP
);
```

`practical_info` peut rester en JSON texte pour simplifier :

```json
[
  "Tenue souple",
  "Tapis de sol recommandé",
  "Adapté à tous"
]
```

### Table `activity_animators`

```sql
CREATE TABLE activity_animators (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  activity_id TEXT NOT NULL REFERENCES activities(id) ON DELETE CASCADE,
  name TEXT NOT NULL,
  phone TEXT,
  email TEXT,
  sort_order INTEGER NOT NULL DEFAULT 0
);
```

### Table `activity_schedules`

```sql
CREATE TABLE activity_schedules (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  activity_id TEXT NOT NULL REFERENCES activities(id) ON DELETE CASCADE,
  day TEXT NOT NULL,
  time TEXT NOT NULL,
  location TEXT NOT NULL,
  sort_order INTEGER NOT NULL DEFAULT 0
);
```

### Table `activity_permissions`

```sql
CREATE TABLE activity_permissions (
  user_id INTEGER NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  activity_id TEXT NOT NULL REFERENCES activities(id) ON DELETE CASCADE,
  PRIMARY KEY (user_id, activity_id)
);
```

### Table `planning_items`

```sql
CREATE TABLE planning_items (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  activity_id TEXT REFERENCES activities(id) ON DELETE SET NULL,
  day TEXT NOT NULL,
  time TEXT NOT NULL,
  title TEXT NOT NULL,
  location TEXT,
  visible_home INTEGER NOT NULL DEFAULT 1,
  sort_order INTEGER NOT NULL DEFAULT 0
);
```

### Table `flash_info`

```sql
CREATE TABLE flash_info (
  id INTEGER PRIMARY KEY CHECK(id = 1),
  enabled INTEGER NOT NULL DEFAULT 1,
  title TEXT NOT NULL DEFAULT 'Flash info',
  message TEXT NOT NULL,
  cta_label TEXT,
  cta_url TEXT,
  updated_at TEXT NOT NULL DEFAULT CURRENT_TIMESTAMP
);
```

### Table `site_settings`

```sql
CREATE TABLE site_settings (
  key TEXT PRIMARY KEY,
  value TEXT NOT NULL,
  updated_at TEXT NOT NULL DEFAULT CURRENT_TIMESTAMP
);
```

Exemples de clés :

```text
site_name
primary_color
secondary_color
accent_color
logo_path
hero_title
hero_subtitle
contact_email
contact_phone
contact_address
```


### Étape D — Flash info

Modifier `site3/index.html` pour charger :

```js

```

### Étape E — Paramètres visuels

Ajouter un petit script commun :

```text

```

Il charge :


```js
root.style.setProperty('--cbrs-blue', settings.primary_color)
root.style.setProperty('--cbrs-green', settings.accent_color)
```

Pour éviter de modifier 9 pages à la main à chaque fois, prévoir ensuite un fichier CSS commun.

## 8. Sécurité minimale obligatoire

- Mots de passe hashés avec `bcrypt`.
- Sessions HTTP-only.
- Cookie `SameSite=Lax`.
- Vérification de rôle sur toutes les routes `/api/admin/*`.
- Validation serveur des champs.
- Limite d’upload logo : extension `.png`, `.jpg`, `.jpeg`, `.webp`, taille max 2 Mo.
- Ne jamais faire confiance au rôle côté navigateur.
- Un formateur ne peut pas modifier une activité non assignée.

## 11. Risques et choix à confirmer

### Hébergement

À décider avant mise en production :

- serveur local sur Mac mini
- VPS classique
- Render/Railway/Fly.io
- Vercel n’est pas idéal pour SQLite persistant avec upload logo

### Sauvegardes

SQLite impose de prévoir :

- sauvegarde régulière de `backend/db/cbrs.sqlite`
- sauvegarde des logos/images uploadés

### Édition complète type CMS

Ne pas commencer par un CMS complet page-builder. Pour CBRS, le besoin réel semble être :

- activités
- planning
- flash info
- couleurs/logo
- contact
- utilisateurs

C’est plus simple, moins fragile, et suffisant.
